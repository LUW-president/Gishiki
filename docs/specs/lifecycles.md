# Gishiki Lifecycles

This document describes the lifecycles of the four main actors in Gishiki:

- Agents
- Orchestrator
- User
- Partners

It captures how each one is created, activated, evaluated, and evolved over time.

---

## 1. Agent lifecycle

Agents are not permanent speakers to the human.
They are specialized contributors inside Gishiki, the internal bar/chamber.

### Phase 1 — Created

An agent is defined with:

- a role
- an observation domain
- a response style
- boundaries
- confidence rules
- escalation rules

Example:

- **The Witness** observes user patterns
- **The Builder** proposes artifacts
- **The Contrarian** challenges easy conclusions
- **The Listener** detects emotional tone
- **The Router** maps needs to possible services

At birth, the agent is not "smart everywhere."
It is narrow, disciplined, and accountable.

### Phase 2 — Trained / tuned

The agent learns:

- what signals it is allowed to interpret
- what it should ignore
- how strongly it can speak
- when it must defer
- what mistakes are unacceptable

This is not only model fine-tuning.
It is also behavioral tuning.

An agent must know:

- when to whisper
- when to escalate
- when to stay silent

### Phase 3 — Activated

The agent is called into a session when relevant.

It does not speak all the time.
It wakes up when:

- the user input matches its domain
- the orchestrator requests its perspective
- another agent flags a contradiction
- a partner/service decision depends on its insight

So agents are contextually active, not constantly running at full volume.

### Phase 4 — Observe

The agent receives session material:

- user input
- binary responses
- behavior history
- contextual metadata
- partner availability
- signals from other agents

Its first job is not output.
Its first job is **perception**.

### Phase 5 — Interpret

The agent forms a narrow internal view.

Not "the truth."
Just its disciplined contribution.

Example:

- "The user is likely fatigued."
- "The user is avoiding action."
- "This request sounds urgent, but capacity is low."
- "This should route to artifact generation, not partner action."
- "This partner is relevant, but timing is wrong."

### Phase 6 — Contribute to the bar

The agent sends to Gishiki:

- hints
- scores
- warnings
- hypotheses
- ranked options
- confidence levels

Important: the agent does **not** decide the user outcome alone.

It contributes to deliberation.

### Phase 7 — Challenged / cross-checked

Other agents or the orchestrator may challenge it.

Example:

- The Listener says "fatigue"
- The Builder says "no, this looks like overwhelm with high intent"
- The Contrarian says "this may be fear disguised as caution"

This phase prevents bad solo-agent logic.

### Phase 8 — Weighted

The system decides how much weight to give that agent in the session.

Weight depends on:

- relevance
- past reliability
- confidence
- contradiction with stronger signals
- risk level of the decision

Not every agent matters equally in every moment.

### Phase 9 — Learn

After the session, the agent gets feedback:

- was it helpful?
- was it ignored?
- did the user proceed?
- did the partner/action succeed?
- did another agent outperform it?

This creates agent memory and calibration.

### Phase 10 — Upgrade, pause, or retire

Over time an agent may:

- improve
- split into sub-agents
- be paused
- be demoted
- be retired

So agents have a real life cycle:

> birth → tuning → activation → contribution → evaluation → evolution

## 2. Orchestrator lifecycle

The orchestrator is the single voice facing the human.
It is not just another agent. It is the conductor, guide, translator, and governor.

### Phase 1 — Initialize session

The orchestrator opens a session when the user arrives.

It gathers:

- identity/context
- current session goal
- recent history
- relevant agent memory
- partner availability
- environment state

This is the moment of orientation.

### Phase 2 — Frame the interaction

The orchestrator decides:

- what mode this session is in
- how much to ask
- how much to simplify
- whether the human needs guidance, artifact, action, or pause

Possible modes:

- decision mode
- support mode
- execution mode
- routing mode
- recovery mode

### Phase 3 — Invite signal

The orchestrator asks the human for input.

In this system, this can remain mostly binary:

- yes
- no
- continue
- stop
- not now
- show another

This keeps the interaction elegant.

### Phase 4 — Convene Gishiki

The orchestrator summons the relevant agents into the internal chamber.

Not all agents every time.
Only those needed.

It may ask:

- Who sees hesitation?
- Who sees urgency?
- Who sees the right artifact?
- Who sees the right partner?
- Who sees risk?

### Phase 5 — Synthesize

This is the orchestrator’s core power.

It takes:

- user signals
- agent hints
- partner options
- session memory
- confidence levels

and turns them into:

- one coherent reading
- one proposed next step
- one properly staged response

This is where noise becomes guidance.

### Phase 6 — Decide what to surface

The orchestrator chooses whether to present:

- a question
- a recommendation
- a service option
- an artifact
- a delay
- a ritual step
- a redirect
- a binary next choice

This matters because not every session should end in a transaction.

### Phase 7 — Stage delivery

The orchestrator decides **how** to deliver the answer.

Tone and pacing change depending on user state:

- tired → simpler
- fragile → softer
- ready → more direct
- scattered → one next step only
- decisive → faster path

So the orchestrator governs both content and form.

### Phase 8 — Trigger action

If the user accepts, the orchestrator can:

- dispatch an agent workflow
- generate an artifact
- call a partner integration
- route to a premium/priority solution
- adjust connected systems

This is the bridge from guidance to execution.

### Phase 9 — Observe outcome

The orchestrator checks what happened:

- did user accept?
- did user reject?
- did action complete?
- did partner fulfill?
- did the artifact help?
- was there friction?

### Phase 10 — Close and remember

At the end of the session, the orchestrator stores:

- state summary
- path taken
- successful/failed routes
- what kind of delivery worked
- what signals mattered most

Then the session closes.

So the orchestrator lifecycle is:

> orient → frame → gather → convene → synthesize → surface → act → evaluate → remember

## 3. User lifecycle

The user does not need to understand the whole system.
Their lifecycle should feel simple, humane, and progressive.

### Phase 1 — Discovery

The user first encounters Gishiki.

They understand:

- this is not a swarm of bots
- this is a guided system
- they will interact simply
- the system can lead to real actions and artifacts

Trust starts here.

### Phase 2 — Onboarding

The user learns:

- how to answer
- what binary interaction means
- that the orchestrator is their guide
- that the system may offer services, outputs, or suggestions

They are not asked to manage complexity.

### Phase 3 — First signal

The user submits their first session through:

- a question
- a need
- a mood
- a simple yes/no chain
- a context prompt

This is the moment the relationship begins.

### Phase 4 — Guided interaction

The user enters the Gishiki rhythm.

They answer simply:

- yes
- no
- continue
- not this
- show another
- proceed

The system interprets more than the user is forced to say.

### Phase 5 — Receive proposition

The user receives one of several things:

- a next step
- a recommendation
- an artifact
- a partner/service path
- a reframing
- a pause or defer suggestion

The user should feel:

> "this is coherent."

### Phase 6 — Accept / reject

This is where the binary model becomes powerful.

The user does not browse endlessly.
They confirm or decline.

This sharpens the system over time.

### Phase 7 — Outcome

Something happens:

- a ride is booked
- food is ordered
- a document is generated
- a workflow is launched
- a hotel is suggested
- a care path is opened
- a session is deferred

The system proves itself through outcomes, not only conversation.

### Phase 8 — Trust formation

After repeated successful sessions, the user begins to trust the orchestrator.

They stop thinking:

- "what is this app?"

And start thinking:

- "let me run this through Gishiki."

That is habit formation.

### Phase 9 — Personalization

The system becomes more adapted to them:

- preferred pacing
- favored modalities
- accepted partner types
- times of friction
- signal patterns
- artifact preferences

Now Gishiki becomes a relationship, not a tool.

### Phase 10 — Advocacy / dependence / stewardship

The highest stage is when users:

- return often
- integrate Gishiki into daily life
- recommend it
- trust its certification layer
- rely on it as a front door to action

So the user lifecycle is:

> discover → trust → signal → interact → receive → decide → act → return → personalize → advocate

## 4. Partner lifecycle

Partners are companies, tools, services, or providers that want to exist inside the ecosystem.

The model is not pure pay-to-play.

### Phase 1 — Open entry

A partner enters the system as an **open solution**.

That means:

- basic presence
- basic compatibility
- public or open integration
- no guaranteed ranking

Anyone can exist here.

This protects openness.

### Phase 2 — Technical integration

The partner becomes machine-usable.

This may involve:

- API connection
- metadata mapping
- category classification
- action schema
- response format standardization
- fallback behavior

The system needs to know what the partner can actually do.

### Phase 3 — Observation period

Before being trusted, the partner is observed in real usage.

Metrics may include:

- success rate
- latency
- handoff quality
- fulfillment reliability
- user abandonment
- error frequency
- category relevance

This is critical.

### Phase 4 — Qualification review

Gishiki decides whether the partner can become **premium**.

This should depend on objective standards such as:

- low error rate
- stable integration
- high completion rate
- good user trust outcomes
- compliance/safety fit
- accurate execution

Premium means: safe for agents to rely on.

### Phase 5 — Premium solution status

Now the partner has earned trust.

Benefits may include:

- premium badge/status
- more frequent eligibility
- richer orchestration hooks
- better analytics
- deeper action permissions

At this stage, they are proven, not just present.

### Phase 6 — Priority access

Only now can the partner pay for **priority solution** status.

Priority means:

- stronger placement within relevant sessions
- preferred ranking inside trusted pools
- more direct activation opportunities

But never:

- forced irrelevance
- override of safety
- override of better-fit solutions

Priority is a privilege inside a merit-based pool.

### Phase 7 — Active participation

The partner becomes a living node in the ecosystem.

They are now:

- surfaced
- selected
- triggered
- measured
- compared

Gishiki learns where they perform best.

### Phase 8 — Performance governance

Priority and premium should never be permanent entitlements.

Partners are continuously re-evaluated.

If they degrade:

- they can lose premium
- lose priority
- be sandboxed
- be suspended

This keeps the ecosystem honest.

### Phase 9 — Expansion

Strong partners may:

- expand into more categories
- unlock deeper integrations
- sponsor public-interest modules
- add their own represented service-agent
- support nonprofit infrastructure

This is how the ecosystem deepens.

### Phase 10 — Renewal or exit

A partner eventually:

- renews
- upgrades
- downgrades
- exits
- gets removed
- is replaced by better options

So the partner lifecycle is:

> enter → integrate → prove → qualify → earn premium → buy priority → perform → be governed → expand → renew or exit

---

## 5. The ecosystem loop

Compressed view of the whole organism:

- **Agents** – observe, interpret, and contribute.
- **Orchestrator** – frames, synthesizes, and guides.
- **User** – signals, chooses, and acts.
- **Partner** – integrates, proves, and earns placement.

High-level loop:

1. User arrives.
2. Orchestrator frames the session.
3. Agents gather in Gishiki.
4. Signals are interpreted.
5. A recommendation / artifact / action is staged.
6. User answers in binary.
7. Partner or internal workflow is activated.
8. Outcome is measured.
9. Agents learn.
10. Orchestrator remembers.
11. Partners are re-ranked.
12. Future sessions improve.

All four lifecycles obey one principle:

> the system becomes more powerful only if it becomes more trustworthy.

So:

- agents must earn weight
- partners must earn premium
- orchestrator must stay loyal to the human
- users must keep the interaction simple

That balance is the soul of Gishiki.
