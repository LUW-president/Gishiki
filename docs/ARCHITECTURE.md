# Gishiki Architecture

## Overview

Gishiki is organized around four layers:

1. **Human layer**
2. **Orchestrator layer**
3. **Agent chamber layer**
4. **Capability network layer**

The system is designed so that the human interacts with only one coherent guide while multiple specialized agents deliberate behind the scenes.

---

## 1. Human layer

This is the visible surface of the system.

The user should experience:

- calm guidance
- simple interaction
- mostly binary decisions
- minimal cognitive load
- one clear next move at a time

Typical user inputs:

- yes
- no
- not now
- continue
- show another
- proceed

The human is never required to manage multiple bots directly.

---

## 2. Orchestrator layer

The orchestrator is the single human-facing intelligence.

Its role is to:

- frame the session
- gather relevant signals
- convene the right internal agents
- synthesize competing perspectives
- choose what to surface
- stage the tone and pacing of delivery
- trigger actions, artifacts, or partner routing

The orchestrator is not a generic chatbot.
It is a governor, translator, and conductor.

---

## 3. Agent chamber layer

Inside Gishiki, specialized agents gather in the internal chamber — the bar.

Each agent has a narrow domain, such as:

- mood or tone detection
- hesitation detection
- contradiction detection
- pattern recognition
- artifact generation
- practical routing
- risk signaling
- partner matching

Agents do not speak directly to the human.
They contribute:

- hints
- warnings
- scores
- ranked options
- interpretations
- confidence levels

The bar exists to create deliberation before output.

---

## 4. Capability network layer

The capability network connects Gishiki to real-world tools and services.

This may include:

- food delivery
- transportation
- lodging
- workplace tools
- connected devices
- care and wellbeing resources
- document or artifact generation systems

A capability is not only a company name.
It is an action the orchestrator can trigger on behalf of the human.

Examples:

- order dinner
- call a ride
- book a room
- generate a draft
- hand a task to a productivity suite
- shift an ambient environment into a different mode

---

## Session flow

A typical session follows this path:

1. User arrives with a need, question, or state.
2. Orchestrator frames the interaction.
3. Relevant agents are activated in the bar.
4. Agents observe and contribute signals.
5. Orchestrator synthesizes the chamber output.
6. Orchestrator proposes one next move.
7. User responds in a simple binary way.
8. If accepted, an internal workflow or external partner action is triggered.
9. Outcome is measured and remembered.

---

## Design principles

### One surface, many minds

The human should experience one guide.
Complexity should stay behind the curtain.

### Relevance before monetization

Partner participation must never override user fit.

### Open ecosystem, governed trust

Any solution may exist in the network.
Only proven solutions may become premium.
Only premium solutions may buy priority placement.

### Action over noise

The point of orchestration is not more talking.
It is a better next move.

---

## Future direction

Over time, Gishiki may evolve from a single orchestrator with hidden agents into a programmable orchestration protocol that other systems can build on.

But the principle should remain constant:

> intelligence gathers before it speaks.
